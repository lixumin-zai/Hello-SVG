在前端开发中，我们有许多图形绘制技术可供选择。了解它们的差异，是做出正确架构决策的前提。

## 1. SVG vs. Canvas

这是前端面试中最经典的对比题。核心差异在于：**SVG 是基于 DOM 的声明式绘图，而 Canvas 是基于像素的命令式绘图**。

<SvgVsCanvas />

**总结：**
*   **用 SVG**：图表（如折线图、饼图）、交互式地图、图标、需要响应鼠标悬停/点击等事件的复杂图形、需要被屏幕阅读器解析的无障碍图形。
*   **用 Canvas**：2D 游戏、大量粒子的特效、实时视频处理、需要绘制数万个数据点的大型散点图。

## 2. SVG vs. WebGL

*   **WebGL**：基于 OpenGL ES 的 JavaScript API，允许在浏览器中进行硬件加速的 3D（和 2D）渲染。它通过 `<canvas>` 元素暴露接口，由 GPU 直接执行。
*   **对比**：
    *   **维度**：SVG 专注于 2D 矢量图形；WebGL 天生为 3D 设计（如 Three.js 等引擎的底层），虽然也能画 2D 但杀鸡用牛刀。
    *   **学习曲线**：SVG 像 HTML 一样容易上手，声明式语法直观；WebGL 需要理解着色器（Shaders）、矩阵变换、顶点缓冲区等极其复杂的计算机图形学概念。
    *   **性能**：WebGL 性能极高，能压榨 GPU 算力，适合渲染超大规模数据或复杂的 3D 游戏场景。

## 3. SVG vs. CSS (Shapes & Clip-path)

CSS 的 `clip-path` 和各种绘制形状的技巧越来越强大，很多时候和 SVG 的应用场景会重叠。究竟用哪个？

<SvgVsCss />

*   **纯 CSS 绘制**：利用 `border-radius`、`box-shadow` 甚至 `linear-gradient` 可以画出很多有趣的形状，但代码往往难以维护（所谓的"CSS 魔法"），且语义不清晰。
*   **SVG 绘制**：更加正规和语义化，代码意图明确，专为图形设计。
*   **结合使用**：如今的最佳实践往往是**用 SVG 绘制路径，用 CSS `clip-path` 引用该路径来裁剪 HTML 元素**，强强联手。

<Callout type="info" title="何时用 CSS，何时用 SVG？">
  如果只是一个简单的圆角、圆形或基础多边形，用 CSS。如果涉及到不规则曲线（贝塞尔曲线）、复杂的内部结构、多图层叠加或复杂的交互动画，毫不犹豫地选择 SVG。
</Callout>

## 5. SVG vs. PDF

有趣的是，SVG 和 PDF 有着深厚的历史渊源。它们本质上都是一种"文档描述语言"，但服务于不同的最终媒介。

*   **SVG 的归宿是屏幕**：它为 Web 交互而生，支持 DOM 脚本、CSS 动画，尺寸随容器自适应。
*   **PDF 的归宿是物理世界**：它为物理媒介（打印机/纸张）而生，有着严格的分页、出血线、色彩空间（CMYK）定义，确保在任何设备上打印输出都分毫不差。

在现代企业级应用中，经常会遇到这样的需求链：前端用 SVG 渲染交互式的数据报表供用户查看分析，后端（或前端借助 html2canvas 等库）将 SVG 页面转换为 PDF 供用户下载归档和打印。它们是互补的技术。
