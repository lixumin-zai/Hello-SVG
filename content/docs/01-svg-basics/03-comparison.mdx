在前端开发中，我们有许多图形绘制技术可供选择。了解它们的差异，是做出正确架构决策的前提。

## 1. SVG vs. Canvas

这是前端面试中最经典的对比题。核心差异在于：**SVG 是基于 DOM 的声明式绘图，而 Canvas 是基于像素的命令式绘图**。

<SvgVsCanvas />

**总结：**
*   **用 SVG**：图表（如折线图、饼图）、交互式地图、图标、需要响应鼠标悬停/点击的复杂图形。
*   **用 Canvas**：2D 游戏、大量粒子的特效、实时视频处理、需要绘制数万个数据点的大型散点图。

## 2. SVG vs. WebGL

*   **WebGL**：基于 OpenGL ES 的 JavaScript API，允许在浏览器中进行硬件加速的 3D（和 2D）渲染。它通过 `<canvas>` 元素暴露接口，由 GPU 直接执行。
*   **对比**：
    *   **维度**：SVG 是 2D 的；WebGL 天生为 3D 设计（如 Three.js），虽然也能画 2D。
    *   **学习曲线**：SVG 像 HTML 一样容易上手；WebGL 需要理解着色器（Shaders）、矩阵变换、顶点缓冲区等复杂的计算机图形学概念。
    *   **性能**：WebGL 性能极高，适合渲染超大规模数据或复杂的 3D 场景。

## 3. SVG vs. CSS (Shapes & Clip-path)

CSS 的 `clip-path` 和各种绘制形状的技巧越来越强大，很多时候和 SVG 的应用场景会重叠。

*   **纯 CSS 绘制**：利用 `border-radius`、`box-shadow` 甚至 `linear-gradient` 可以画出很多有趣的形状，但代码往往难以维护（所谓的"CSS 魔法"）。
*   **SVG 绘制**：更加正规和语义化，代码意图明确。
*   **结合使用**：如今的最佳实践往往是**用 SVG 绘制路径，用 CSS `clip-path` 引用该路径来裁剪 HTML 元素**。

<Callout type="info" title="何时用 CSS，何时用 SVG？">
  如果只是一个简单的圆角、圆形或基础多边形，用 CSS。如果涉及到不规则曲线（贝塞尔曲线）、复杂的内部结构和多图层叠加，毫不犹豫地选择 SVG。
</Callout>

## 4. SVG vs. PNG/JPEG/WebP

虽然它们都是用来在网页上显示"图像"的，但定位完全不同：

| 格式 | 类型 | 适用场景 | 缩放表现 |
| :--- | :--- | :--- | :--- |
| **SVG** | 矢量图 | 图标、Logo、UI 插图、图表 | 无损放大，永远清晰 |
| **PNG** | 光栅图 (无损压缩) | 包含透明背景的复杂图像、UI 切图（传统方式） | 放大变模糊 |
| **JPEG** | 光栅图 (有损压缩) | 摄影照片、色彩丰富的背景图 | 放大变模糊，有压缩伪影 |
| **WebP** | 光栅图 (现代) | 替代 PNG/JPEG 的全能格式，体积更小 | 放大变模糊 |

## 5. SVG vs. PDF

有趣的是，SVG 和 PDF 有着深厚的历史渊源（还记得 PGML 吗？）。它们本质上都是一种"文档描述语言"。

*   **SVG 的归宿是屏幕**：它为 Web 交互而生，支持 DOM 脚本、CSS 动画。
*   **PDF 的归宿是打印机**：它为物理媒介（纸张）而生，有着严格的分页、出血线、色彩空间（CMYK）定义，确保在任何打印机上输出都分毫不差。

在现代企业级应用中，经常会遇到这样的需求链：前端用 SVG 展示数据报表，后端（或前端）将 SVG 转换为 PDF 供用户下载打印。它们是互补的技术。
