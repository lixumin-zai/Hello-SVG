## 终极魔法：`matrix(a, b, c, d, e, f)`

SVG 的 2D 变换使用一个 3x3 的齐次变换矩阵。由于最后一行永远是 `[0, 0, 1]`，所以我们在代码中只需要写前两行的 6 个参数：

| 参数 | 含义 |
| :--- | :--- |
| **a** | X 轴缩放 (Scale X) |
| **b** | Y 轴倾斜 (Skew Y) |
| **c** | X 轴倾斜 (Skew X) |
| **d** | Y 轴缩放 (Scale Y) |
| **e** | X 轴平移 (Translate X) |
| **f** | Y 轴平移 (Translate Y) |

所有的基础变换最终都会转换成这 6 个数字。比如 `translate(10, 20)` 就等于 `matrix(1, 0, 0, 1, 10, 20)`。
在自动化工具（如 SVGO）优化 SVG 代码时，经常会把多个连续的 `transform` 合并成一个简短的 `matrix` 以节省体积。

## 组合变换：顺序决定命运

当你把多个变换写在同一个 `transform` 属性中时（例如 `transform="translate(100) rotate(45)"`），有一个极其反直觉的规则：
**浏览器是从右向左（或从内向外）执行这些变换的！**

通过上面的演示，你可以看到：
- **先平移，后旋转（情况A）**：旋转时是以原来的 `(0,0)` 原点为中心，所以平移出去的图形会被甩出一个弧度。
- **先旋转，后平移（情况B）**：旋转改变了局部坐标系的角度，随后的平移是沿着倾斜后的新坐标轴行进的。

<Callout type="warn" title="分组变换 (g 标签)">
  当你使用 <code>&lt;g&gt;</code> 标签嵌套时，顺序依然遵循这个规则：<br/>
  <code>&lt;g transform="translate(10)"&gt;&lt;g transform="scale(2)"&gt;...&lt;/g&gt;&lt;/g&gt;</code> <br/>
  内层的 <code>scale(2)</code> 会先执行，外层的 <code>translate(10)</code> 后执行。
</Callout>
