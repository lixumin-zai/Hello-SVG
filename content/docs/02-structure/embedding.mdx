SVG 作为图片格式，有着极高的灵活性。但这也带来了选择困难症：**我该用什么方式把它放到网页上？**

不同的嵌入方式，决定了你对 SVG 的控制能力（CSS 样式、JS 交互、缓存等）。

## 交互实验：嵌入方式实验室

下面这个实验室展示了 4 种最常用的 SVG 嵌入方式。点击不同的选项，看看它们的优缺点和代码差异。

## 2.2.1 内联 SVG (Inline SVG) —— 最强大

这是现代 Web 开发中最推荐的方式，特别是当你需要**交互**或**动态修改样式**时。

```html
<!-- 直接写在 HTML 里 -->
<svg class="icon" viewBox="0 0 24 24">
  <path d="..." />
</svg>
```

- **优点**：
  - CSS 可以直接控制内部元素的颜色 (`fill: currentColor`)、大小、动画。
  - JS 可以直接操作 DOM，绑定事件。
  - 没有额外的 HTTP 请求（但也无法缓存）。
- **缺点**：
  - HTML 体积变大。
  - 无法跨域复用（除非用 `<use>`）。

## 2.2.2 `<img>` 标签 —— 最简单

就像插入 JPG/PNG 一样简单。

```html
<img src="icon.svg" alt="图标" />
```

- **优点**：
  - 浏览器会自动缓存文件。
  - 语义清晰，自带 `alt` 属性。
- **缺点**：
  - **无法修改内部样式**（如 hover 变色）。
  - **无法运行脚本**（出于安全考虑）。
  - 交互性差。

## 2.2.3 CSS `background-image` —— 纯装饰

适用于装饰性的背景图案。

```css
.icon {
  background-image: url("icon.svg");
}
```

- **优点**：
  - HTML 结构干净。
  - 可以利用 CSS Sprite 技术（虽然现在少用了）。
- **缺点**：
  - 同样无法修改内部样式。
  - 无障碍性差（屏幕阅读器读不到）。

## 2.2.4 `<object>` 与 `<iframe>` —— 历史遗留

在 HTML5 普及之前，为了获得交互性，人们常用 `<object>` 或 `<embed>`。

```html
<object data="chart.svg" type="image/svg+xml"></object>
```

- **优点**：
  - 可以缓存。
  - 内部脚本可以运行（同源策略允许的话）。
  - 有回退内容（fallback）。
- **缺点**：
  - 交互略显笨重（需要 `contentDocument` 获取内部 DOM）。
  - 存在一些兼容性怪癖。

## 2.2.5 Data URI —— 性能优化

将 SVG 转为 Base64 编码，直接嵌入 CSS 或 HTML 中。

```css
.icon {
  background-image: url("data:image/svg+xml;base64,...");
}
```

- **优点**：
  - 减少 HTTP 请求。
- **缺点**：
  - Base64 会增加约 33% 的体积。
  - 无法缓存（除非放在 CSS 文件里一起缓存）。
  - 难以阅读和调试。

## 总结建议

1.  **图标系统**：推荐 **内联 SVG**（配合 React 组件或 Sprite）。
2.  **大图/插画**：推荐 **`<img>` 标签**（利用缓存）。
3.  **背景纹理**：推荐 **CSS background**。
4.  **交互式图表**：推荐 **内联 SVG**（D3.js）。
